## Phase 3: Handoff Protocols

### 3.0 Agent Relationship Model

All agents are **peers** — no agent is a sub-process of another. The user or slash command orchestrates agent transitions, not agents calling each other.

| Relationship           | Direction                 | Mechanism                                                   | Example                                   |
| ---------------------- | ------------------------- | ----------------------------------------------------------- | ----------------------------------------- |
| **Design handoff**     | designer → site-builder   | User copies brief, switches agent                           | Screenshot → Design Brief → page build    |
| **Quality gate**       | site-builder → code       | Site-builder finishes, user invokes code for lint/typecheck | Page built, needs lint verification       |
| **Feature pipeline**   | architect → code → review | Slash command `/feature` routes sequentially                | New API endpoint implemented and reviewed |
| **Cross-layer change** | architect flags in design | Architect notes "site-builder needed after code"            | Core API change requires template updates |

**Key rule:** Code is a **quality gate** for site-builder output — it validates (lint, typecheck, format) but never produces site-layer code. Code does NOT hand off to site-builder. If a core code change (packages/) requires site template updates (demos/), the architect must flag it in the design. The user invokes site-builder separately.

```
Peer Agent Map:

designer ──(design brief)──▶ site-builder ──(quality gate)──▶ code ──(review)──▶ review
                                │
architect ──(contracts)────────▶ code ──(review)──────────────────▶ review
                                │
plugin-builder ──(quality gate)─▶ code ──(review)─────────────────▶ review
```

Site-builder and plugin-builder both produce code that passes through code's quality gate. Code itself produces core code that passes through review. No agent delegates work to another — each transition is a sequential handoff orchestrated by the user or a slash command.

### 3.1 Feature Work Protocol

```
User describes feature
         │
         ▼
   ┌─────────────┐
   │  architect   │  Reads codebase, analyzes dependency graph
   │ (V4 Pro,     │  Produces design: contracts, data flow, file list
   │  no-thinking)│
   └──────┬───────┘
          │ handoff: architect writes structured design in conversation
          ▼
   ┌─────────────┐
   │    code      │  Implements per architect's design
   │  (V4 Flash)  │  Writes code + tests, runs lint:quick, typecheck, format
   └──────┬───────┘
          │ handoff: code signals "implementation complete, ready for review"
          ▼
   ┌─────────────┐
   │   review     │  Adversarial review using EmDash checklist
   │  (V4 Flash)  │  Returns: bugs by severity, file:line, fix
   └──────┬───────┘
          │
          ├── NO BUGS → PR ready (add changeset if needed)
          │
          └── BUGS FOUND → code fixes critical/high → review re-checks
```

**Handoff format from architect to code:**

```
## Design

### Affected Files
- packages/core/src/... (new/modify)
- packages/admin/src/... (new/modify)

### Contracts
- New type: `Foo` in `types.ts`
- API endpoint: `POST /_emdash/api/foo` returns `{ success, data }`

### Data Flow
[Mermaid or text description]

### Edge Cases to Handle
- Empty results → 200 with empty items[]
- Duplicate slug → 409 CONFLICT
- Missing locale → 400 BAD_REQUEST

### Migration Required?
- Yes → NNN_description.ts with up/down
- No

### Tests Needed
- Unit: handler returns ApiResponse<Foo>
- Integration: e2e flow through route
```

### 3.2 Bug Fix Protocol (TDD)

```
User reports bug
         │
         ▼
   ┌─────────────┐
   │    code      │  1. Write failing test that reproduces bug
   │  (V4 Flash)  │  2. Implement fix
   └──────┬───────┘  3. Verify test passes
          │          4. Run lint:quick, typecheck, format
          │
          ├── Simple fix → PR ready (skip review)
          │
          └── Complex/risky fix → handoff to review
                     │
                     ▼
              ┌─────────────┐
              │   review     │  Adversarial review
              └─────────────┘
```

**Bug fix is "complex/risky" if:** involves SQL/migrations, auth changes, data integrity, or touches >3 files.

**Cross-layer awareness (Q2 — audit finding):** Bug fixes skip the architect, so no one reviews the fix for downstream impact. Before completing a bug fix, code must check whether the change alters the signature or behavior of any public API used by site-layer code in `demos/` or `templates/`. If so, code must warn the user: _"This fix changes the behavior of `[API]`. Templates in `[demos/X]` may need updating — consider invoking site-builder."_ The review agent also checks for this during the optional review step.

### 3.3 PR Review Protocol (External)

```
User provides PR number or URL
         │
         ▼
   ┌─────────────┐
   │   review     │  1. Fetch PR diff via GitHub MCP
   │  (V4 Flash)  │  2. Read all changed files
   └──────────────┘  3. Apply adversarial review checklist
                     4. Post review comments
```

### 3.4 Routine Review Protocol (Uncommitted Changes)

```
User invokes /review on working tree
         │
         ▼
   ┌─────────────┐
   │   review     │  1. Run git diff to see uncommitted changes
   │  (V4 Flash)  │  2. Read all modified files
   └──────────────┘  3. Apply adversarial review checklist
                     4. Return findings
```

### 4.1 New Slash Commands

Create these in `.kilo/command/`:

#### `/feature` (feature.md)

```markdown
---
description: Implement a feature through architect → code → review pipeline
---

Route feature work through the full pipeline:

1. **architect** — analyze $ARGUMENTS, read relevant code, produce a design (contracts, data flow, migration plan, test plan)
2. **code** — implement per architect's design, write tests, run lint:quick+typecheck+format
3. **review** — adversarial review of all changes
4. Fix critical/high bugs, then signal PR-ready with changeset if needed
```

#### `/fix` (fix.md)

```markdown
---
description: Fix a bug using TDD — write failing test, fix, verify
---

Fix a bug described in $ARGUMENTS:

1. Write a failing test that reproduces the bug
2. Implement the minimal fix
3. Verify the test passes
4. Run `pnpm --silent lint:quick`, `pnpm typecheck`, `pnpm format`
5. If the fix touches SQL/migrations/auth or >3 files, invoke `review` agent
```

#### `/review-pr` (review-pr.md)

```markdown
---
description: Adversarial review of an external GitHub PR
agent: review
---

Review the GitHub PR specified in $ARGUMENTS:

1. Fetch PR diff using `gh pr diff $ARGUMENTS`
2. Read all changed files in context
3. Apply the adversarial review checklist from AGENTS.md:
   - Logic correctness (locale filters, id usage, pagination)
   - Edge cases (empty/null/missing/o)
   - State & concurrency (TOCTOU, stale closures)
   - SQL & security (sql.raw(), json_extract, CSRF, auth)
   - Data integrity (indexes, forward-only migrations)
   - i18n & RTL (Lingui, logical Tailwind)
   - Resource management (O(n²), chunking, after(fn))
4. Post findings as review comments on the PR
```

#### `/pr` (pr.md)

```markdown
---
description: Create a GitHub pull request with the EmDash PR template
---

Create a pull request:

1. Run `pnpm --silent lint:json | jq '.diagnostics | length'` — stop if not clean
2. Run `pnpm typecheck` — stop if failing
3. Add a changeset if the change affects a published package (`pnpm changeset --empty`)
4. Push the current branch
5. Create the PR using the template from `.github/PULL_REQUEST_TEMPLATE.md`
6. Fill out all sections of the template
```

#### `/review` (review.md) — project command

```markdown
---
description: Adversarial review of current working tree changes
agent: review
---

Review uncommitted changes in the working tree:

1. Run `git diff` to see all changes
2. Read every modified file in full context
3. Apply the adversarial review checklist from AGENTS.md
4. Report bugs ordered by severity (CRITICAL → HIGH → MEDIUM → LOW)
5. If no bugs found, say "No bugs found"
```

## Phase 5: When to Use Site-Builder vs Plugin-Builder vs Code

### Decision Tree

```
Task description includes:
  ├── "page", "template", "seed", "collection schema",
  │   "Astro", "Portable Text", "menu", "taxonomy",
  │   "widget", "deploy", "demos/", "templates/"
  │   → Use site-builder (loads building-emdash-site + emdash-cli skills)
  │
  ├── "plugin", "hook", "sandbox", "block kit",
  │   "definePlugin", "plugin descriptor", "plugin admin",
  │   "packages/plugins/"
  │   → Use plugin-builder (loads creating-plugins skill)
  │
  ├── Everything else:
  │   "packages/core/", "packages/admin/", "packages/auth/",
  │   "migration", "API route", "handler", "kysely",
  │   "database", "auth", "middleware"
  │   → Use code (follows AGENTS.md constraints)
```

### WordPress Migration Special Cases

- `wordpress-theme-to-emdash` skill — invoke via **site-builder** (it builds Astro pages from WP themes)
- `wordpress-plugin-to-emdash` skill — invoke via **plugin-builder** (it ports WP plugins to EmDash plugins)

## Phase 8: EmDash MCP Server Connection

### 8.1 Background

EmDash includes a built-in MCP server at `/_emdash/api/mcp` (`packages/core/src/mcp/server.ts`) that exposes **33 tools** across seven domains:

| Domain   | Tools                                                                                                                                                                                                                                                                                                           | Count |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----- |
| Content  | `content_list`, `content_get`, `content_create`, `content_update`, `content_delete`, `content_restore`, `content_permanent_delete`, `content_publish`, `content_unpublish`, `content_schedule`, `content_compare`, `content_discard_draft`, `content_duplicate`, `content_list_trashed`, `content_translations` | 15    |
| Schema   | `schema_list_collections`, `schema_get_collection`, `schema_create_collection`, `schema_delete_collection`, `schema_create_field`, `schema_delete_field`                                                                                                                                                        | 6     |
| Media    | `media_list`, `media_get`, `media_update`, `media_delete`                                                                                                                                                                                                                                                       | 4     |
| Search   | `search`                                                                                                                                                                                                                                                                                                        | 1     |
| Taxonomy | `taxonomy_list`, `taxonomy_list_terms`, `taxonomy_create_term`                                                                                                                                                                                                                                                  | 3     |
| Menu     | `menu_list`, `menu_get`                                                                                                                                                                                                                                                                                         | 2     |
| Revision | `revision_list`, `revision_restore`                                                                                                                                                                                                                                                                             | 2     |

**Transport:** Streamable HTTP in **stateless mode** (`POST /_emdash/api/mcp`). No SSE or session management needed.

**Authentication:** Three methods available:

1. **OAuth 2.1 Authorization Code + PKCE** — Standard MCP client flow. User approves scopes in browser.
2. **Personal Access Token (PAT)** — Long-lived `ec_pat_*` tokens from the admin panel. Simplest for dev.
3. **Device Flow** — CLI-style flow (`emdash login`). Used for headless environments.

**Scopes:**
| Scope | Grants |
|-------|--------|
| `content:read` | List, get, compare, search content. List taxonomies, menus. |
| `content:write` | CRUD content, publish/unpublish, schedule, duplicate, restore. Create taxonomy terms. |
| `media:read` | List, get media items. |
| `media:write` | Update, delete media metadata. |
| `schema:read` | List collections, get collection schemas. |
| `schema:write` | Create/delete collections and fields. |
| `admin` | Full access (implies all scopes). |

**OAuth Discovery:**

- `GET /.well-known/oauth-protected-resource` — Resource metadata with auth server URL and supported scopes
- `GET /.well-known/oauth-authorization-server/_emdash` — Auth server metadata with grant types, endpoints

### 8.2 Auth Method Decision

For Kilo's agent-based workflow connecting to a local EmDash instance:

| Method                | Pros                                 | Cons                                                | Verdict                        |
| --------------------- | ------------------------------------ | --------------------------------------------------- | ------------------------------ |
| **PAT**               | Simple, no browser, permanent        | Must create in admin UI first, stored as secret     | ✅ Primary (dev)               |
| **OAuth Device Flow** | No manual token creation, CLI-native | Requires `emdash login`, needs browser for approval | ⚠️ Secondary / deployed sites  |
| **OAuth Code + PKCE** | MCP-standard, auto-discovery         | Needs browser redirect, complex for headless CLI    | ❌ Not suitable for CLI agents |

**Decision:** PAT for local dev (agents connect to `localhost:4321`), Device Flow for deployed/staging sites. Both are supported via the same `Bearer` token header.

### 8.3 Connection Prerequisites

Before configuring Kilo, the user must:

1. **Have a running EmDash instance** — `cd demos/simple && pnpm dev` (local) or a deployed URL
2. **Create a PAT in the admin UI** — Settings → API Tokens → Create token with required scopes
3. **Know the base URL** — `http://localhost:4321` (local dev) or `https://example.com` (deployed)

### 8.4 Kilo MCP Configuration

Add a `remote` MCP server entry to `kilo.json`:

```jsonc
{
	"$schema": "https://app.kilo.ai/config.json",
	"mcp": {
		"github": {
			/* existing */
		},
		"emdash": {
			"type": "remote",
			"url": "http://localhost:4321/_emdash/api/mcp",
			"headers": {
				"Authorization": "Bearer ${EMDASH_PAT}",
			},
			"enabled": true,
		},
	},
	"skills": {
		/* existing */
	},
}
```

**Key configuration points:**

- `type`: `"remote"` — Streamable HTTP transport
- `url`: The full MCP endpoint URL. Kilo appends `.well-known` discovery paths automatically if the server supports OAuth discovery, but for PAT auth the URL is used directly.
- `headers` or `auth`: The PAT in `Authorization: Bearer <token>` header
- Secret management: Use `${EMDASH_PAT}` environment variable reference (or `auth.token` with env var)

**Alternative auth configurations:**

For **OAuth device flow** (deployed sites):

```jsonc
"emdash": {
  "type": "remote",
  "url": "https://example.com/_emdash/api/mcp",
  "enabled": true
  // Kilo auto-discovers OAuth via /.well-known/oauth-protected-resource
}
```

For **inline PAT** (quick dev, NOT committed):

```jsonc
"emdash": {
  "type": "remote",
  "url": "http://localhost:4321/_emdash/api/mcp",
  "headers": {
    "Authorization": "Bearer ec_pat_abc123..."
  },
  "enabled": true
}
```

### 8.5 Scope Recommendations by Agent

Different agents need different MCP tool access. Recommended scope assignments:

| Agent            | Recommended Scopes                             | Rationale                                            |
| ---------------- | ---------------------------------------------- | ---------------------------------------------------- |
| `site-builder`   | `content:read`, `schema:read`, `media:read`    | Reads collections, content, and media to build pages |
| `plugin-builder` | `content:read`, `content:write`, `schema:read` | Tests plugin hooks by reading/writing content        |
| `review`         | `content:read`                                 | Reviews content structure and queries                |
| `code`           | `content:read`, `schema:read`                  | Verifies handler behavior against real content       |
| `architect`      | `schema:read`, `content:read`                  | Analyzes schema structure (read-only)                |

**For simplicity in dev**, a single PAT with `content:read,content:write,schema:read,media:read` covers all agents. Avoid `schema:write` and `admin` unless actively modifying schema via MCP.

````markdown
---
description: Connect Kilo agents to an EmDash MCP server
---

Connect Kilo to an EmDash site's MCP server so agents can interact with content programmatically.

## Prerequisites

1. A running EmDash instance (local: `cd demos/simple && pnpm dev`, or deployed URL)
2. A Personal Access Token (PAT) from the admin UI at Settings → API Tokens
3. The base URL of your EmDash instance

## Steps

1. **Verify the MCP server is reachable:**
   ```bash
   curl -X POST $BASE_URL/_emdash/api/mcp \
     -H "Authorization: Bearer $EMDASH_PAT" \
     -H "Content-Type: application/json" \
     -d '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
   ```
````

Should return 33 tools.

2. **Add the MCP server to `kilo.json`:**
   Add under `mcp`:

   ```jsonc
   "emdash": {
     "type": "remote",
     "url": "$BASE_URL/_emdash/api/mcp",
     "headers": {
       "Authorization": "Bearer $EMDASH_PAT"
     },
     "enabled": true
   }
   ```

3. **Set the environment variable:**

   ```bash
   export EMDASH_PAT="ec_pat_..."
   ```

4. **Verify connection:**
   The MCP server will be available to all agents. Test with:
   ```
   /audit-sql — or any command that uses the MCP tools
   ```

## Scopes

For development, create a PAT with: `content:read`, `content:write`, `schema:read`, `media:read`.

For production/deployed sites, use more restrictive scopes per agent.

```

### 8.8 Known Limitations

- **No SSE streaming** — Stateless mode only. Long-running operations (imports, rebuilds) won't stream progress.
- **No media upload via MCP** — The MCP server doesn't expose upload tools (media files are uploaded through the admin UI or REST API).
- **PAT must be created before connecting** — No self-service token creation via MCP.
- **Localhost dev bypass not available** — The MCP server requires real auth (PAT or OAuth). The `dev-bypass` only works for session cookies, not MCP Bearer tokens.
- **Single instance** — Each `kilo.json` connects to one EmDash instance. Multi-site workflows need separate Kilo projects.

### 8.9 MCP Usage by Agent (When to Leverage EmDash Tools)

After connection, each agent should use MCP tools in specific scenarios. When MCP is not available, agents fall back to static analysis:

#### Site-builder

| MCP Tool | When to Use | Fallback (no MCP) |
|----------|------------|-------------------|
| `schema_list_collections` | Verify what collections exist live | Read `seed/seed.json` and `emdash-env.d.ts` |
| `schema_get_collection` | Confirm field names, types, constraints | Read `seed/seed.json` collection schema |
| `content_list` | Verify seed content was loaded, check available content | `npx emdash seed` + check server logs |
| `content_get` | Fetch a specific entry to verify its shape | Read `seed/seed.json` content section |
| `taxonomy_list` | Confirm taxonomy names before building archive pages | Read `seed/seed.json` taxonomies section |
| `search` | Test search results during development | Navigate to `/search?q=test` in browser |
| `menu_get` | Verify navigation menu structure | Read `seed/seed.json` menus section |

**Pattern:** Site-builder first attempts MCP for live schema discovery. If MCP is unavailable (server down, no PAT), falls back to `seed/seed.json` and `emdash-env.d.ts` for static schema information. Static fallback works for schema discovery but cannot verify that seed data was loaded successfully.

#### Code

| MCP Tool | When to Use | Fallback (no MCP) |
|----------|------------|-------------------|
| `schema_get_collection` | Verify handler behavior against live schema | Read `seed/seed.json` + unit test against in-memory SQLite |
| `content_list` | Test content queries during handler development | Unit test with test database fixtures |
| `content_create` / `content_update` | Test write operations via MCP | Unit test handler functions directly |

**Pattern:** Code uses MCP for integration verification (real DB). Without MCP, relies on unit tests with `better-sqlite3` in-memory databases, which is the project's standard testing pattern.

#### Review

| MCP Tool | When to Use | Fallback (no MCP) |
|----------|------------|-------------------|
| `content_list` | Verify locale filters work correctly | Read handler code + seed schema statically |
| `search` | Test search queries return expected results | Read search implementation + index config |
| `schema_get_collection` | Verify index coverage by cross-referencing schema | Read migration files + schema registry code |

**Pattern:** Review primarily works statically against code. MCP adds data integrity verification against live content. Static fallback is the default mode for review — MCP is an enhancement, not a requirement.

#### Architect and Designer

- **Architect**: MCP for live schema analysis. Fallback: reads `seed/seed.json`, migration files, and schema registry code statically. Architect already operates in read-only-first mode regardless of MCP availability.
- **Designer**: Does not use MCP tools at all. Works exclusively from screenshots. No fallback needed.

### 8.10 Integration with Handoff Protocols

| Protocol | MCP Role |
|----------|----------|
| **Feature work** (architect → code → review) | Architect reads schema via MCP. Code verifies handlers via MCP. Review checks data integrity via MCP. |
| **Bug fix** (code → review) | Code uses MCP to verify the fix against real content if available. |
| **Visual design** (designer → site-builder → code → review) | Site-builder reads schema and content via MCP before implementing the Design Brief. |
| **PR review** (review) | Review fetches PR diff via GitHub MCP, then optionally validates queries via EmDash MCP if the site is available. |
```

### 9.7 Designer → Site-Builder Handoff Rules

The handoff between `designer` and `site-builder` follows these rules:

1. **Designer is read-only** — it does not write Astro files, CSS, or configuration. The site-builder does.
2. **User is the courier** — designer produces the brief → user reviews → user copies brief → user switches to site-builder agent → pastes the brief.
3. **Brief is the contract** — site-builder implements exactly what the brief specifies. If something is ambiguous, site-builder asks the user (not the designer — the designer's session ended).
4. **When the brief is missing details** — site-builder fills gaps with sensible EmDash defaults (e.g., Base layout from template, standard Kumo tokens for unspecified colors).
5. **After implementation** — site-builder hands off to `code` for lint/typecheck/format, then optionally `review` for visual correctness check.
